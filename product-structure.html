Dhrubok Portal Structure

Complete functional layout mapping the four distinct operational experiences, access rules, and new Convex & Clerk backend concepts.

Dhrubok now separates coaching workflows into four distinct experiences tailored to specific roles, keeping students and teachers contained within clear operational sandboxes while owners manage the entirety of coaching center tasks.

1. Public Coaching Website

The public site implements marketing-level designs featuring spacious layout templates, strong dark CTA pills, course showcases, and dynamic responsive layouts for mobile users.

Main Pages

The public route structure houses the following surfaces:

  • Home: Coaching introduction and primary admissions trigger.
  • Courses: List of topics, descriptions, and structures.
  • Teachers: Public teacher profiles (independent of internal records).
  • Notices: General coaching announcements.
  • About & Contact: Location map, hours, history, and contact details.
  • Admission: Public admission application form.
  • Portal Sign-In: Access point to student, teacher, or owner portals.

Homepage Specifications

The landing page captures prospective students and handles standard branding items:

  • Spacious hero intro with primary Apply for admission CTA.
  • Interactive lists of active courses and upcoming/open batches.
  • Coaching accomplishments, stats strip, and featured teacher cards.
  • Important notice alerts and contact info.
  • Clear login link for enrolled students and active teachers.

Content Management (CMS)

Owners govern public website content via the Owner Portal. Content states support draft, preview, and published phases to prevent internal records updates from instantly showing up publicly.

2. Public Admission Application

Applicants submit admission requests directly without creating an account first. The system processes applicants through a multi-step checklist, finishing with SMS notification and profile creation.

Applicant-Provided Information

Forms collect both structural student parameters and guardian details:

  • Student’s names (Bangla and English) + optional phone.
  • Guardian details (Name, Phone number, and family relationship).
  • School or college name + current class.
  • Preferred course and batch selection (with disclaimer explaining batch selections may change).
  • Optional profile photo and additional note.

Review Panel & Settings

The Owner portal handles application processing. The review panel displays submitted fields next to owner-defined operational fields like ID generation and discounts.

3. Clerk Authentication Decision

Dhrubok uses Clerk for authenticating users while retaining roles and fine-grained access rules inside Convex databases.

How it works:

Convex reads Clerk credentials to identify a user. Access roles are checked on each query or mutation rather than relying on browser-provided user claims. This ensures database-level security.

Convex & Clerk Integration Model

Recommended Student Sign-In Workflow

Since students might not have unique emails, they log in using their generated Student ID as their Clerk username. First-time log-ins require a password change.

  • Create student record → Generate unique ID (e.g. std-1024).
  • Assign initial one-time temporary password.
  • Send access credentials to the guardian via SMS.
  • Prompt student to configure a private password upon first sign-in.

Portal Capabilities

  • Routine Viewer: Displays class sessions across enrolled batches. Conflicting routines are flagged.
  • Attendance tracker: Shows summaries of class presences and absences.
  • Fees Ledger: Transparent billing statements, discounts, and downloadable receipts.
  • Exams & Results: Shows exam reports only after results are published by owners.
  • Profile Self-service: Update low-risk details (photo, email, phone) directly. Updates to critical fields (enrolment, class, fees) require owner approval.

Teacher Permissions & Guardrails

To keep coaching records clean and secure, teachers operate under structured guardrails:

  • Assigned Batches only: Roster access is restricted to classes they teach.
  • Marks Entry: Teachers enter scores, but owners must review and approve them before they are published to students or sent via SMS.
  • Materials & Notices: Can upload class files and send targeted batch alerts. Global notices require owner credentials.

6. Owner Portal Additions

The Owner portal handles administration, admission request queues, profile change requests, user permissions, and website content controls.

Core Dashboard Features

  • Application inbox: Filter, evaluate, and admit public applicants.
  • User Controls: Enable/disable portals, reset passwords, and manage teacher profiles.
  • Website CMS: Edit courses, banners, notices, and teacher bio fields.
  • Change Request approvals: Approve or reject requests for sensitive student profile updates (e.g. guardian phone numbers).

8. New Data Structure

The following tables and data structures support the multi-portal experience in Convex. Authentication credentials and user roles remain separate from operational profiles.

portalAccounts
  - clerkUserId: string (Clerk identity mapping)
  - role: "owner" | "teacher" | "student"
  - linkedProfileId: id (owner, teacher, or student table link)
  - status: "active" | "suspended" | "pending"
  - passwordResetRequired: boolean

admissionApplications
  - studentNameBangla: string
  - studentNameEnglish: string
  - guardianName: string
  - guardianPhone: string
  - guardianRelationship: string
  - preferredCourseId: id
  - preferredBatchId: id
  - status: "new" | "review" | "accepted" | "rejected"
  - reviewerId: id (linked owner ID)

studentProfileChangeRequests
  - studentId: id
  - requestedChanges: object (photo, address, phone updates)
  - status: "pending" | "approved" | "rejected"
  - reviewedBy: id

siteContent
  - sectionKey: string (e.g. "homepage.hero")
  - contentBangla: string
  - contentEnglish: string
  - state: "draft" | "published"

materials
  - teacherId: id
  - batchId: id
  - fileUrl: string
  - state: "draft" | "published"

notices
  - authorId: id
  - scope: "global" | "course" | "batch"
  - targetId: id (optional course/batch link)
  - sendSmsTriggered: boolean

9. Portal Routing

Dhrubok routes web traffic into three primary sandboxes based on user roles checked by Convex mutations and Clerk middleware.

/ — Public website landing page
/admission — Online admission form
/sign-in — Clerk auth page
/owner/* — Admin tools & financial logs
/teacher/* — Attendance & marks manager
/student/* — Student progress logs & receipts

Important Security Notice:

Frontend route redirects do not guarantee data isolation. Every query and mutation inside Convex must check the authenticated Clerk session to verify permissions.

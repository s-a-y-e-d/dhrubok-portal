Make Courses the control room

A repo-grounded redesign brief for making owner/courses faster, clearer, and harder to misconfigure. The recommendation is not to add a large LMS. It is to turn each course into a compact operational hub that guides an owner from setup to enrolment, teaching, publishing, and completion.

Executive summary

Core diagnosis: Dhrubok has the right academic entities and safe backend rules, but the owner must mentally assemble a course from separate tabs and screens. The system exposes its data structure instead of the owner’s goal: “open this course and get it ready to run.”
1

Primary destination

A searchable course list with status, readiness, batch capacity and next action.

5

Course workspace sections

Overview, Batches, Subjects & teachers, Schedule, Public page.

3

Delivery phases

Fix navigation first, then operational signals, then time-saving automation.

Top five moves

  1. Replace the nested generic record browser with a course-first list and detail workspace.
  2. Add a setup/readiness checklist and a single contextual “next action”.
  3. Show operational facts on every course row: batches, active students, seats, subject coverage, next class, public/admission state.
  4. Unify course creation, subject linking, first-batch setup and publishing into a save-and-continue flow.
  5. Add duplicate/roll-forward for a new academic session; do not force owners to rebuild recurring structure.

Course list

Default landing page for owner/courses. Desktop uses a compact table; mobile uses record cards. Columns: Course, Session, Status, Batches, Students / capacity, Setup, Next class, Public / admissions, Actions. Search should match Bangla name, English name and code.

Course workspace

A selected course opens a stable detail route or URL-addressable pane. Its header carries name, code, session, lifecycle badge and a single primary next action. Sections should be:

  • Overview: readiness, active batches, active students, seat utilisation, next class, missing setup and recent changes.
  • Batches: batch list, capacity, admission state, roster count and “create batch”.
  • Subjects & teachers: course subjects first; per-batch assignment coverage beneath each subject.
  • Schedule: routine summary and conflicts, scoped to this course, with batch filter.
  • Public page: bilingual preview, cover, descriptions, public visibility, admissions availability and order.

Readiness model

Do not use an arbitrary percentage. Use named checks with links: identity complete, at least one subject, at least one active/planned batch, teacher coverage, routine coverage, fees configured where applicable, public copy complete, admission-open batch available. Label the state Needs setup, Ready to run, or Completed; show the missing items in text.

Prioritized recommendations

P0

Course-first list and persistent workspace

Make Courses the default view and keep the course context while the owner moves between batches, subjects, assignments, schedule and publishing.

Implementation direction: dedicated list query returning aggregate counts and next class; URL such as ?courseId=…&view=overview; retain existing editors behind the new structure initially.

P0

Visible readiness and exact blockers

Show what is missing before activation, publication or archive. Archive should list blocking batches with links; publication should explain that only active courses can be public and identify missing bilingual content or open batches.

P0

Progressive creation

Step 1 captures session, Bangla name, English name and code. Generate a slug suggestion from English name/code; create as draft. Step 2 offers “Add subjects”, “Create first batch”, or “Finish later”. Publishing content becomes a later task, not a gate to saving the academic record.

P1

Operational course summary

Add active student count, batch count, total capacity, seats left, teacher/routine gaps, next scheduled class and public admission state. Every metric should drill into its contributing records.

P1

Duplicate / roll forward to a new session

Offer a preview that copies selected metadata, subjects and optionally batch templates/routines. Never copy enrolments, attendance, payments, exams or already-created class sessions. Require new dates and review teacher assignments before activation.

P1

Publishing preview and reorder

Put academic and website readiness in one course workspace. Replace manual sort-order entry with Move up / Move down controls (or accessible reordering later). Show the exact Bangla and English public card/detail preview before publication.

P2

Cross-course calendar and conflict intelligence

Add schedule views by course, batch, teacher and room, with conflict warnings before save. This follows the strongest Teachworks/TutorCruncher pattern but should come after course navigation and readiness are fixed.

What to change, add, remove and keep

Change

  • Default landing from Operations to Courses.
  • Generic nested tabs into a course list + workspace.
  • Session’s silent first-option default into an explicit “current session” filter.
  • Raw status words into localized labels and semantic badges.
  • Archive rejection into blocker preview and resolution links.

Add

  • Search and status/session filters.
  • Draft, active, completed and archived visibility.
  • Readiness checklist and contextual next action.
  • Counts, capacity, next class and setup gaps.
  • Duplicate/roll-forward preview.
  • Unsaved-change guard and save feedback.

Remove / demote

  • Separate “Create New” as a top-level destination; use New course and contextual create actions.
  • Manual public sort number from the main form.
  • Slug as a mandatory first-decision; auto-suggest with advanced edit.
  • Website publishing controls duplicated across unrelated screens.
  • Repeated English fallbacks in Bangla views.

Keep

Keep the session → course → batch hierarchy, bilingual fields, public/private separation, immutable operational records, dependency-safe archive, audit logging and server-side validation. These are sound foundations.

Recommended first build slice

Build the course-first list plus overview shell before redesigning every editor. It delivers immediate usability, creates the right information architecture, and lets the existing forms continue working while sections are migrated incrementally.

How to know it worked

  • Median clicks from owner/courses to editing a known course: target ≤ 2.
  • Time to create a draft and first batch: target under 2 minutes for an experienced owner.
  • Percentage of active courses with no unexplained setup gaps: target 100%.
  • Archive/publish actions rejected only after submit: target near zero; blockers should be known beforehand.
  • New-session setup time and manual field re-entry after roll-forward: reduce by at least 50%.
  • Mobile: every primary action and record card meets the 44px target; no bilingual text clipping.
